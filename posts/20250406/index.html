10 Smart console Methods to Improve Your JavaScript Debugging #

Let‚Äôs go beyond console.log() and explore more powerful tools for smarter debugging.

Most developers rely on console.log() during development, and it‚Äôs often the first step when checking whether values are correct or code is functioning properly. However, the console object provides many more useful methods that can make debugging more efficient and your code more organized.

Here are 10 powerful console methods every JavaScript developer should know:

1. console.error() #

Highlights error messages in red. Use it when you want to make critical issues or exceptions stand out.

console.error("An error occurred!");

2. console.warn() #

Displays warning messages in yellow. Ideal for non-critical issues users should be aware of.

console.warn("This is a warning!");

3. console.info() #

Outputs informational messages. Often similar in appearance to console.log(), depending on the browser.

console.info("This is an informational message.");

4. console.table() #

Displays arrays or objects in a nice tabular format. Especially helpful for inspecting JSON data.

const users = [
  { name: "Chulsoo", age: 25 },
  { name: "Younghee", age: 30 }
];
console.table(users);

5. console.dir() #

Outputs an interactive list of an object‚Äôs properties. Useful for inspecting HTML elements or deep object structures.

const obj = document.body;
console.dir(obj);

6. console.group() / console.groupEnd() #

Groups related logs together in a collapsible format, making nested logs easier to follow.

console.group("User Info");
console.log("Name: Chulsoo");
console.log("Age: 25");
console.groupEnd();

7. console.time() / console.timeEnd() #

Measures how long a block of code takes to run ‚Äî useful for performance monitoring.

console.time("Execution Time");
for (let i = 0; i < 1000000; i++) {}
console.timeEnd("Execution Time");

8. console.assert() #

Logs a message only if the condition is false.

const value = 1;
console.assert(value === 2, "Value is not equal to 2!");

9. console.clear() #

Clears the console output.

console.clear();

10. console.count() / console.countReset() #

Counts the number of times a specific label has been called.

function greet(name) {
  console.count("Greet Count");
  console.log(`Hello, ${name}!`);
}

greet("Chulsoo");
greet("Younghee");
console.countReset("Greet Count");
greet("Minsu");

Bonus: console.trace() #

Prints a stack trace to show the code path taken to reach the current point.

console.trace("Trace call");

Why You Should Clean Up console Logs in Production #

Leaving too many console logs in production code can lead to:

  1. Performance issues due to excessive rendering in developer tools.
  2. Security concerns from leaked data like user info or internal logic.
  3. Messy code that‚Äôs harder to maintain and review.
  4. Larger bundle sizes due to unnecessary logs.

Best Practices to Remove Console Logs Before Deployment #

‚úÖ Use Babel Plugin #

Install a plugin to automatically remove console logs during production builds:

npm install --save-dev babel-plugin-transform-remove-console

Then configure it in .babelrc:

{
  "env": {
    "production": {
      "plugins": ["transform-remove-console"]
    }
  }
}

‚úÖ Use ESLint #

Warn or error when console.log() is used:

// .eslintrc.js
module.exports = {
  rules: {
    "no-console": "warn" // or "error"
  }
};

‚úÖ Create a Custom Logger #

Use a wrapper to conditionally log messages based on the environment:

  log: (...args) => {
    if (process.env.NODE_ENV !== "production") {
      console.log(...args);
    }
  }
};
import { logger } from './logger';
logger.log("Debugging message");

‚úÖ Static Analysis Tools #

Use tools like eslint, sonarqube, husky, prettier, or even grep:
